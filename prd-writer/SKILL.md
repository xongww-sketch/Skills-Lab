# PRD Writer

为研发团队输出清晰、可执行的产品需求文档。核心原则：**多表多图少废话，研发看完就能干。**

## 工作流程

### 第一步：收集信息

从用户提供的材料中提取关键信息：
- 聊天记录、会议纪要、口头描述均可作为输入
- 主动读取用户发送的文件（Excel模板、截图等），提取隐含规则
- 信息不足时追问，但不超过3个问题/轮

必须明确的核心要素：
1. **业务背景** — 为什么要做这件事
2. **数据来源** — 数据从哪来，格式是什么
3. **处理逻辑** — 怎么处理（匹配、转换、校验）
4. **输出目标** — 最终产物是什么，给谁用
5. **约束条件** — 限制、规则、异常场景

### 第二步：规划文档结构

根据需求复杂度选择章节（不是每个都要，按需裁剪）：

| 章节 | 何时需要 | 内容 |
|---|---|---|
| 背景 | 必须 | 业务背景 + 现状问题 + 要做什么（用表格，不写长段落） |
| 数据模型 | 有数据映射时 | 实体关系、映射表、字段说明 |
| 规则约束 | 有外部限制时 | 从模板/系统中提取的硬性规则（如数量上限、格式要求） |
| 核心流程 | 必须 | 每个主流程单独一节，含流程图 + 取数规则表 |
| 防重复/幂等 | 有重复风险时 | 重复场景 × 对策矩阵 |
| 外部系统对接 | 涉及RPA/API/第三方时 | 操作步骤表 + 前提条件 + 异常处理 |
| DB设计 | 需要持久化时 | 表结构（字段/类型/说明），突出追溯能力 |
| 整体架构 | 多系统协作时 | 架构图（生成图片插入） |
| 待确认项 | 必须 | 未确认问题 × 确认人，表格形式 |

### 第三步：撰写文档

#### 格式铁律

1. **表格 > 文字** — 能用表格说清的绝不写段落
2. **图片 > 代码块** — 流程图、架构图、泳道图必须生成图片，不用ASCII
3. **每节独立可读** — 研发可以只看自己相关的章节
4. **数据示例必须有** — 用真实数据格式举例（如Excel填写示例）
5. **异常场景必须覆盖** — 每个流程都要有"如果XX失败怎么办"

#### 写作风格

- 目标读者是研发，不是老板 — 写具体实现，不写商业价值
- 用确定性语言 — "必须"、"不能"、"当XX时YY"，不用"建议"、"可以考虑"
- 字段/表名/状态值用 `code` 格式标注
- 每个规则都标明来源（"抖音模板限制" vs "我们自己的设计"）

### 第四步：生成图片

流程图和架构图用 HTML 渲染 + Playwright 截图，插入飞书文档。

生成流程：
1. 写 HTML 文件（参考 `references/diagram-style.md` 的样式规范）
2. `npx playwright screenshot --full-page --viewport-size="1200,800" file:///tmp/diagram.html /tmp/diagram.png`
3. `feishu_doc upload_image` 上传到文档指定位置（用 `index` 参数控制位置）

图片设计原则：
- 用彩色圆角卡片表示步骤，不同角色/阶段用不同颜色
- 箭头连接，从左到右或从上到下
- 底部加关键规则卡片（如约束条件、异常处理）
- 字体用系统字体：`-apple-system, "PingFang SC", "Microsoft YaHei", sans-serif`
- 宽度 1200px，背景白色

### 第五步：输出到飞书

1. `feishu_doc create` 创建文档
2. `feishu_doc write` 写入 Markdown 内容（注意：Markdown 表格不支持，会自动转为飞书表格）
3. 上传图片到正确位置（先写内容，再按 block index 插入图片）
4. 发送链接给用户

## 质量检查清单

输出前自检：
- [ ] 每个流程都有流程图（图片，非代码块）
- [ ] 所有硬性规则都提取并标注来源
- [ ] 异常场景全部覆盖
- [ ] 待确认项列出具体问题和确认人
- [ ] 数据示例用真实格式
- [ ] 没有大段落文字（超过3行的段落考虑拆成表格）

## 文件说明

- `references/diagram-style.md` — 流程图/架构图的 HTML 样式规范
- `references/example-structure.md` — 一个完整 PRD 的章节结构示例
